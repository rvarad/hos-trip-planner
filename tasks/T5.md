# T5: Geo provider stack + routing integration

**Goal:** Record the geo stack in an ADR and replace `build_legs`' interim
haversine internals with real OSRM road routing — per-leg distance, duration, and
geometry — behind the unchanged `build_legs` contract.

**Chosen stack (pure OSS, no API key):**
- Map rendering: **MapLibre GL JS** with free vector tiles (T5a).
- Geocoding + autocomplete + reverse: **Photon** (public `photon.komoot.io` now,
  self-host later via `PHOTON_URL`); proxied by Next.js (T7).
- Routing: **OSRM** (public demo `router.project-osrm.org` now, self-host later via
  `OSRM_URL`).

## Design (agreed 2026-06-01)

- **HTTP client:** Python **stdlib `urllib.request` + `json`** with a timeout —
  no new dependency. Plenty for a couple of best-effort GETs.
- **OSRM call:** one GET per leg —
  `{OSRM_URL}/route/v1/driving/{lng},{lat};{lng},{lat}?overview=full&geometries=geojson`.
  Parse `routes[0]`: `distance` m → miles, `duration` s → minutes (round),
  `geometry.coordinates` `[lng,lat]` → our `(lat, lng)` tuples.
- **Failure → graceful fallback + flag:** on any OSRM error (network, timeout,
  non-200, no route), fall back to the existing haversine `_leg` for that leg. The
  trip-level `routing` flag is `"osrm"` if every leg came from OSRM, else
  `"estimated"`. The app never hard-fails; the response tells the frontend when
  distances are approximate.
- **The haversine code from T4 becomes the permanent fallback** (no longer interim).
- **Public-server caveat:** the OSRM demo has no SLA and forbids heavy traffic;
  fine to build/demo against. T12 self-hosts behind `OSRM_URL`.
