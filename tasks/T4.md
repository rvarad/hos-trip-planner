# T4: DRF POST /api/plan-trip

**Goal:** Expose the HOS engine over HTTP. `POST /api/plan-trip` accepts the trip
form's three resolved locations plus cycle hours and start time, builds the route
legs, runs `plan_trip`, and returns the JSON timeline.

## Design (agreed 2026-06-01)

Reconciles the old location-triple request idea with the leg-based engine model.

- **Request contract:** the body is the **three locations** (`current_location`,
  `pickup`, `dropoff`) + `cycle_hours_used` + `start_time_minutes`. The Django
  view turns locations → legs by routing (backend = compute hub; the browser only
  talks to the Next.js proxy).
- **Routing seam:** routing lives in a new `backend/hos/routing.py`. T4 ships an
  **interim `build_legs`** that estimates each leg by straight-line **haversine**
  distance and an assumed average speed (`AVG_SPEED_MPH = 55`), with a degenerate
  two-point geometry. The endpoint works end-to-end and is unit-testable with no
  network. **T5 swaps the internals for OSRM** (real road distance + geometry);
  the `build_legs` signature and the view stay unchanged.
- **Hours → minutes** conversion happens here, at the input boundary (not in the
  engine): `current_cycle_used_minutes = round(cycle_hours_used * 60)`.
- **Conventions:** serializers for every request/response shape — no hand-built
  dicts in the view (CLAUDE.md). The view is thin and delegates to `routing` and
  `engine`. The engine stays pure; the HTTP/routing I/O lives in the view/routing.
- `days` serializes as `[]` until T9. Route **geometry for the map** is deferred
  to T5/T10; segments already carry point locations for markers.

## Subtasks

- [x] **T4.1** `LocationSerializer`
  - Fields: `label` (CharField, non-empty), `lat` (FloatField, [-90, 90]),
    `lng` (FloatField, [-180, 180]).
  - Test: a valid location validates; empty label and out-of-range lat/lng each
    raise `ValidationError`.

- [ ] **T4.2** `PlanTripRequestSerializer`
  - Nests `LocationSerializer` for `current_location`, `pickup`, `dropoff`; plus
    `cycle_hours_used` (FloatField, 0–70) and `start_time_minutes` (IntegerField,
    0–1439, default 0).
  - Test: a full nested payload validates; a missing location and an out-of-range
    `cycle_hours_used` are each rejected.

- [ ] **T4.3** Response serializers
  - `DutySegmentSerializer` (`start_min`, `end_min`, `status` as its string value,
    `description`, nested `start_location`/`end_location`, `miles`),
    `DayLogSerializer` (`date_offset`, `segments`), and `PlanResultSerializer`
    (`segments`, `days`, `total_miles`).
  - Test: serializing a sample `PlanResult` yields the expected nested dict
    (status rendered as e.g. `"driving"`, locations as `{label, lat, lng}`).

- [ ] **T4.4** Interim leg builder (`backend/hos/routing.py`)
  - `build_legs(current, pickup, dropoff) -> list[RouteLeg]`: two legs
    (current→pickup, pickup→dropoff). Distance via haversine; `duration_minutes =
    round(distance_miles / AVG_SPEED_MPH * 60)`; `geometry` = the two endpoints.
  - Pure (no network); `math` only — no new dependencies.
  - Test: returns two legs; a known coordinate pair gives the expected haversine
    distance (within tolerance) and the matching duration. T5 will replace internals.

- [ ] **T4.5** `PlanTripView` + URL wiring
  - `POST /api/plan-trip`: validate request → `build_legs` → convert
    hours→minutes → build `TripInput` → `plan_trip` → serialize `PlanResult`.
    Wire the route into `config/urls.py` (via `hos/urls.py`).
  - Test (DRF `APIClient`): a valid body returns 200 with the timeline shape; an
    invalid body returns 400.
