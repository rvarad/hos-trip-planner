# T6: Next.js MUI input form

**Goal:** The trip planner page: a top form bar (three `<LocationField>` pickers +
cycle hours + start time + a Plan button) over a full-width map, wired to submit a
trip to `/api/plan-trip` and surface the result.

## Design (agreed 2026-06-01)

- **Layout:** a **top form bar** above a full-width `<MapView>`. The bar is a
  responsive flex-wrap container (one row when wide, wraps when narrow). Note: the
  `<LocationField>` buttons make each field wide — converting "Drop pin" /
  "Use my location" to compact icon buttons is a T11 polish item (needs an icons dep).
- **Inputs → API shape (T4):** three `LocationField`s (`current_location`,
  `pickup`, `dropoff`) resolving to `{ label, lat, lng }`; `cycle_hours_used`
  (number 0–70); `start_time_minutes` from a native `TextField type="time"`
  (default `08:00` → 480; `"HH:MM"` → minutes on submit). No new dependency.
- **Why start time is collected:** it doesn't affect the engine (HOS math is
  relative minutes), but it anchors the ELD logs to a real clock. A 7am start
  must render as off-duty 00:00–07:00 then duty from 07:00 — not driving from
  midnight. T8 must render unaccounted grid time as off-duty so the pre-start
  hours appear correctly.
- **Map feedback:** as locations resolve, the map shows their markers
  (`MapView markers`, fit-to-bounds). The route polyline + stop markers are **T10**.
- **Submit + result:** POST the assembled body to `/api/plan-trip` (T7.1); store
  the response. Show a **minimal summary** (total miles, day/segment count, and an
  "approximate distances" banner when `routing === "estimated"`). The rich output —
  route map (**T10**) and ELD log sheets (**T8**) — consumes the stored result later.
- **State:** the page (`app/page.tsx`, client) owns the three location values +
  cycle hours + start time + the plan result; `MapView` is dynamically imported
  (`ssr: false`).
- **Testing:** mock `LocationField`, `MapView`, and `fetch` so the form's state,
  validation, and submit body are tested in isolation.
