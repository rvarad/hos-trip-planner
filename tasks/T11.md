# T11: Polish & UI redesign

**Goal:** Turn the working-but-rough UI into a sleek, Uber/Ola-style app:
a left command panel + hero map, icon-based controls, a map legend, in-map
pin picking, loading/error states, and a responsive mobile layout. Done on the
`ui-redesign` branch.

## Design decisions (agreed 2026-06-02)

- **Layout — Option 1:** a fixed **left command panel** (~360px, dark surface)
  beside a **full-bleed hero map**. The panel holds the title, the three location
  fields (stacked, with Uber-style connector dots), compact cycle/start inputs, a
  full-width Plan button, and a results summary card.
- **ELD logs — Map ⇄ Daily Logs toggle:** a toggle over the right pane swaps the
  map for a **full-width** stack of daily log sheets (the 24h grids need the
  horizontal room), instead of squeezing them under the map.
- **Markers:** per-kind **icons** (fuel pump, bed/rest, coffee/break, flags for
  pickup/drop-off, dot for current) plus a **map legend**.
- **Drop pin — tap-to-place on the main map:** tapping a field's 📍 icon *arms*
  that field; the hero map shows a crosshair + hint banner, the next map tap
  drops the marker and reverse-geocodes it, and the marker stays draggable. This
  replaces the separate modal map (poor UX — redundant second map, lost context).
  (Center-pin "pan & confirm" style is a possible later upgrade.)
- **Form-bar clutter (original T6.1 issue):** resolved by the panel layout +
  icon buttons.
- **Map information (agreed 2026-06-02):** surface itinerary detail on the map,
  on-demand to stay clean. All of the below use data already in the response
  (`segments`: status/description/location/miles + `start_min` → clock via
  `start_time_minutes`); **no backend change needed**. (The "remaining HOS at
  each stop" idea is deferred — it would need the engine to expose per-segment
  clock snapshots.)
  - *Quick wins:* marker popups (type, arrival clock + Day N, duration,
    cumulative miles), a trip-summary overlay card, origin/destination labels.
  - *Medium:* color the route by day (+ day-start markers); link Map ⇄ Daily
    Logs (selecting a day highlights that day's route segment + stops).

- [ ] **T11.12** Color route by day
  - Render the route in per-day colors (or place day-start markers) so the

- [x] **T11.14** Pin/geolocation as dropdown options + stale-plan handling
  - Moved "Drop a pin on the map" / "Use my location" out of the input into
    distinct (accent, icon) options at the top of the search dropdown; the input
    is now a clean field (no adornment gap). Full label shown as a caption.
  - State handling: a plan is a snapshot keyed by its inputs (`inputsKey`).
    Map pins always derive from the live inputs; when the inputs drift from the
    plan, the plan is shown **stale** — dimmed/dashed route + a "re-plan to
    update" notice — instead of silently showing outdated markers (option 2).

- [x] **T11.15** Camera policy — focus-newest while entering, fit on Plan
  - `MapView` no longer fits-to-all on every marker change. It `flyTo`s a
    `focusPoint` (the most recently set/edited location) and fits all markers
    only when `fitSignal` bumps (on a successful Plan). Stops the "zoom out to
    include both" feeling when adding a second location.

- [ ] **T11.13** Link Map ⇄ Daily Logs
  - Selecting a day in the Daily Logs view highlights that day's route segment +
    its stops on the map (and ideally vice-versa), tying the two views together.
